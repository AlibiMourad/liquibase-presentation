Datenbank-konfigurations-management mit Liquibase

Software-Configuration-Management

SCM hat mehrere Ziele:

  • Definition und Verfolgung von Prozessen
  • Dokumentation aller Vorgänge
  • Versionierung und Konfliktbehandlung
  • Verwaltung von Voraussetzungen
  • Effizienzsteigerungen bei der automatisierten Applikationserstellung
  • Integration aller vorhandenen Werkzeuge
  • Zugriffskontrolle

Wikipedia zu Software-Configuration-Management

Warum überhaupt Datenbankkonfigurationsmanagement?

  • Datenbankschema ist Code
  • Single Source of Truth
  • Änderungsverfolgung
  • Automatische Migration und Test (z. B. im Rahmen von Continuous Integration)
  • (Halb-) Automatisches Rollback

Warum Werkzeugunterstützung?

  • "Geordnete Evolution"
  • Ein Haufen (inkrementeller) SQL-Skripte skaliert nicht
  • Datenbank Refactoring
  • SQL-Skripte manuell ausführen?
  • Menschliche Interaktion minimieren

Warum gerade Liquibase?

  • Aktives Projekt
  • Umfangreiches Feature-Set
  • Einfach erweiterbar
  • Maven-Plugin
  • Spring-Integration
  • Hibernate-Integration
  • DBDoc - JavaDoc für Datenbanken

Welche Gründe sprechen dagegen?

  • Zeitaufwand für Einlernen in Liquibase
  • Abhängigkeit von einem weiteren Projekt
  • XML statt zusätzlich zu SQL
  • Doppelte Pflege des Datenbankschemas (?)
  • Disziplin notwendig - keine manuellen Änderungen mehr
